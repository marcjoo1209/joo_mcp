# 05. LangGraph 로 엔티티 기반 의도 파악

> 대상: "LLM이 알아서 도구를 고르는 것"과 "명시적으로 의도를 파악해 처리하는 것"의
> 차이를 이해하고 싶은 사람

## 왜 또 다른 경로(`/intent`)인가?

이 프로젝트에는 자연어 CRUD 경로가 둘 있습니다.

| 경로 | 방식 | 특징 |
|---|---|---|
| `/chat` | Gemini 가 **직접** 도구를 선택(함수 호출 루프) | 유연함. 다만 "왜 그 도구를 골랐는지"는 모델 내부 |
| `/intent` | **LangGraph** 가 단계별로 처리 | **엔티티/의도가 겉으로 드러남**. 예측·관찰 가능 |

`/intent` 는 "엔티티를 뽑아 의도를 파악한다"는 과정을 코드로 명시합니다.
그래서 어떤 엔티티를 인식했고 어떤 의도로 분류했는지 응답에서 그대로 볼 수 있습니다.

---

## LangGraph 란?

상태(state)를 들고 **노드(node)** 들을 **엣지(edge)** 로 이어 실행하는 그래프 오케스트레이션 도구입니다.
- 각 노드는 그냥 함수입니다. 상태를 입력받아 일부를 갱신해 돌려줍니다.
- 상태가 노드를 따라 흐르며 점점 채워집니다.

이 프로젝트의 그래프:

```
   START
     │
     ▼
 extract_entities   ← [LLM] Gemini 구조화 출력으로 엔티티 추출
     │                {action, note_id, title, content}
     ▼
 classify_intent    ← [규칙] 엔티티로 의도 확정 (+보정)
     │                "create" / "read_one" / "read_all" / "update" / "delete" / "unknown"
     ▼
   execute          ← [MCP] 의도에 맞는 도구를 결정론적으로 호출
     │                create_note / list_notes / get_note / update_note / delete_note
     ▼
   respond          ← [템플릿] 결과를 한국어 답변으로 (LLM 미사용)
     │
     ▼
    END
```

> 핵심: **LLM 호출은 `extract_entities` 한 곳뿐**입니다. 나머지(분류·실행·응답)는
> 전부 결정론적이라 같은 입력이면 항상 같은 동작을 합니다. → 비용↓, 예측성↑

## 상태와 노드 (`app/services/intent_graph.py`)

```python
class IntentState(TypedDict, total=False):
    message: str                  # 입력
    entities: ExtractedEntities   # extract_entities 가 채움
    intent: str                   # classify_intent 가 채움
    result: Any                   # execute 가 채움
    reply: str                    # respond 가 채움
```

```python
builder = StateGraph(IntentState)
builder.add_node("extract_entities", self._extract_entities)
builder.add_node("classify_intent", self._classify_intent)
builder.add_node("execute", self._execute)
builder.add_node("respond", self._respond)
builder.add_edge(START, "extract_entities")
builder.add_edge("extract_entities", "classify_intent")
builder.add_edge("classify_intent", "execute")
builder.add_edge("execute", "respond")
builder.add_edge("respond", END)
graph = builder.compile()
```

### 1) 엔티티 추출 — Gemini 구조화 출력
응답을 자유 텍스트가 아니라 **정해진 스키마(JSON)** 로 받습니다.

```python
response = await client.aio.models.generate_content(
    model=settings.gemini_model,
    contents=EXTRACTION_PROMPT.format(message=state["message"]),
    config=types.GenerateContentConfig(
        response_mime_type="application/json",
        response_schema=ExtractedEntities,   # ← Pydantic 스키마 강제
    ),
)
entities = response.parsed                    # ExtractedEntities 인스턴스
```

### 2) 의도 분류 — 규칙 + 보정
LLM 없이 엔티티만으로 의도를 확정하고, 모자란 정보는 보정합니다.

```python
# 예) read 인데 id 없으면 전체 조회로, id 가 필요한데 없으면 unknown 으로
if action == "read_one" and e.note_id is None:
    action = "read_all"
if action in ("read_one", "update", "delete") and e.note_id is None:
    action = "unknown"
if action == "create" and not e.title:
    action = "unknown"
```

### 3) 실행 — 의도 → 특정 MCP 도구 (결정론적)
`/chat` 은 모델이 도구를 고르지만, 여기서는 **의도가 곧 도구**입니다.

```python
if intent == "create":   result = await call("create_note", {...})
elif intent == "read_all": result = await call("list_notes", {})
elif intent == "read_one": result = await call("get_note", {"note_id": e.note_id})
...
```

### 4) 응답 — 템플릿
결과를 정해진 한국어 문장으로 변환합니다(추가 LLM 호출 없음).

## 사용 예

```bash
curl -X POST http://127.0.0.1:8000/intent \
  -H "Content-Type: application/json" \
  -d "{\"message\":\"회의록 이라는 제목으로 메모 만들어줘\"}"
```

응답(의도/엔티티가 그대로 드러남):
```json
{
  "intent": "create",
  "entities": { "action": "create", "note_id": null, "title": "회의록", "content": null },
  "result": { "id": 1, "title": "회의록", "content": "", "...": "..." },
  "reply": "메모를 생성했습니다. (id=1)"
}
```
