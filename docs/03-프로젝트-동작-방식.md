# 03. 이 프로젝트 동작 방식 (코드 따라 읽기)

> 대상: 표준 레이어드 아키텍처에서 각 계층이 어떻게 맞물리는지 이해하고 싶은 사람

앞 문서의 개념이 코드 어디에 있는지, 그리고 요청이 계층을 어떻게 흐르는지 짚어봅니다.

---

## 계층 구조 한눈에 보기

요청은 **위에서 아래로만** 의존합니다. (아래 계층은 위 계층을 모릅니다.)

```
        ┌──────────────────────────────────────────────┐
HTTP →  │ api/routes/   (Presentation) 얇은 컨트롤러       │
        │   notes.py · chat.py                           │
        └───────────────────┬──────────────────────────┘
                            │ Depends (의존성 주입: api/deps.py)
        ┌───────────────────▼──────────────────────────┐
        │ services/     (Business) 비즈니스 규칙           │
        │   note_service.py · chat_service.py            │
        └───────────────────┬──────────────────────────┘
                            │
        ┌───────────────────▼──────────────────────────┐
        │ repositories/ (Data Access) SQL 캡슐화          │
        │   note_repository.py                           │
        └───────────────────┬──────────────────────────┘
                            │
        ┌───────────────────▼──────────────────────────┐
        │ db/           (Infrastructure) 연결/초기화       │
        │   database.py → notes.db                       │
        └──────────────────────────────────────────────┘

  가로지르는 것들: core/config(설정) · core/exceptions(도메인 예외)
                  models(도메인 모델) · schemas(입출력 DTO)
```

| 계층 | 폴더 | 책임 | "무엇을 모르나" |
|---|---|---|---|
| Presentation | `app/api/` | HTTP 입출력, DTO 변환 | 비즈니스 규칙·SQL |
| Business | `app/services/` | 규칙·흐름·오케스트레이션 | HTTP·SQL |
| Data Access | `app/repositories/` | SQL 쿼리 | HTTP·비즈니스 규칙 |
| Infrastructure | `app/db/` | DB 연결 | 그 외 전부 |

이렇게 나누면 **한 계층을 바꿔도 다른 계층이 안 흔들립니다.**
(예: SQLite → PostgreSQL 교체는 Repository/db 만 수정.)

---

## 두 갈래의 요청 흐름

같은 Service/Repository 를 **사람(REST)과 AI(MCP)가 공유**합니다.

```
[A] 직접 REST :  POST /notes ─► notes.py ─► NoteService ─┐
                                                          ├─► NoteRepository ─► notes.db
[B] AI 기반   :  POST /chat ─► chat.py ─► ChatService ─► Gemini ─► MCP 서버 ─┘
                                                       (mcp_server/server.py 도 NoteRepository 사용)
```

→ AI가 만든 메모를 `GET /notes`로 즉시 확인할 수 있는 이유입니다.

---

## 1단계 — 도구를 정의한다 (`mcp_server/server.py`)

```python
mcp = FastMCP("notes")
repo = NoteRepository()                 # ★ REST와 동일한 Repository 재사용

@mcp.tool()
def create_note(title: str, content: str = "") -> dict:
    """새 메모를 생성한다. ...(생략)..."""
    return asdict(repo.create(title, content))
```

- `@mcp.tool()` : 이 함수를 **AI가 호출 가능한 도구**로 등록.
  - **함수 이름** → 도구 이름 / **타입 힌트** → 인자 스키마 / **docstring** → 도구 설명
- 도구는 직접 SQL 을 짜지 않고 **Repository 에 위임**합니다.
  → 사람(REST)과 AI(MCP)의 데이터 접근 규칙이 한 곳에만 존재.

> 💡 docstring 은 **AI를 위한 설명서**입니다. 명확할수록 AI가 도구를 잘 고릅니다.

## 2단계 — 서버에 연결하고 Gemini에 넘긴다 (`app/services/chat_service.py`)

```python
self._server_params = StdioServerParameters(
    command=sys.executable, args=[settings.mcp_server_path],   # server.py 실행
)

async with stdio_client(self._server_params) as (read, write):   # ① 프로세스 실행
    async with ClientSession(read, write) as session:            # ② 세션(Client)
        await session.initialize()                                # ③ 핸드셰이크
        response = await self._client.aio.models.generate_content(
            model=settings.gemini_model,
            contents=message,
            config=types.GenerateContentConfig(
                system_instruction=SYSTEM_INSTRUCTION,
                tools=[session],                                  # ★ 마법의 한 줄
            ),
        )
```

- ①~③ : `python mcp_server/server.py`를 하위 프로세스로 띄우고 stdio 로 연결. 이 `session`이 [02번 문서]의 **Client**.
- `tools=[session]` : `google-genai` SDK 가 [02번 문서]의 3~7단계(도구 목록 조회 → 호출 → 결과 전달)를 **자동 수행**.
  → 우리가 function-calling 루프를 직접 돌리지 않음.

## 3단계 — HTTP로 노출한다 (`app/api/routes/`)

### (A) AI 경로 — `chat.py`

```python
@router.post("/chat", response_model=ChatResponse)
async def chat(body: ChatRequest, service: ChatService = Depends(get_chat_service)):
    reply = await service.handle(body.message)     # 2단계로 위임
    return ChatResponse(reply=reply)
```

### (B) 직접 경로 — `notes.py`

```python
@router.post("", response_model=NoteOut, status_code=201)
def create_note(body: NoteCreate, service: NoteService = Depends(get_note_service)):
    note = service.create_note(body.title, body.content)   # Service 에 위임
    return NoteOut.from_model(note)
```

라우터는 **얇습니다**. HTTP 입출력과 DTO 변환만 하고, 로직은 Service 에 맡깁니다.
`Depends(...)` 는 `app/api/deps.py` 의 제공자를 통해 Service 를 **주입**받습니다(직접 `new` 하지 않음).

---

## 4단계 — 예외를 HTTP로 변환한다 (`core/exceptions.py` + `main.py`)

Service 는 HTTP 를 모르므로, 없는 메모는 도메인 예외로 표현합니다.

```python
# services/note_service.py
def get_note(self, note_id: int) -> Note:
    note = self.repository.get(note_id)
    if note is None:
        raise NoteNotFoundError(note_id)      # HTTP가 아닌 도메인 예외
    return note
```

```python
# main.py — 웹 계층에서만 HTTP를 안다
@app.exception_handler(NoteNotFoundError)
async def _note_not_found(request, exc):
    return JSONResponse(status_code=404, content={"detail": str(exc)})
```

→ 비즈니스 로직과 웹 계층이 깔끔히 분리됩니다.

## 5단계 — 모든 것을 조립한다 (`app/main.py`)

```python
def create_app() -> FastAPI:
    app = FastAPI(..., lifespan=lifespan)   # lifespan: 시작 시 init_db()
    app.include_router(notes.router)
    app.include_router(chat.router)
    # 예외 핸들러 등록 ...
    return app

app = create_app()
```

`main.py`는 **조립만** 하는 곳(Composition Root)입니다.

## 직접 실험해볼 것

1. **새 도구 추가** — `repositories/note_repository.py`에 `search(keyword)` 추가 →
   `mcp_server/server.py`에 `@mcp.tool() search_notes` 노출. 재시작 후 `/chat`에
   "장보기 들어간 메모 찾아줘". **라우터 수정 없이 AI가 새 도구를 씁니다.**
2. **저장소 교체** — `db/database.py`와 Repository 만 PostgreSQL 용으로 바꿔보세요.
   Service·Router·MCP 도구는 손대지 않아도 됩니다. (계층 분리의 효과 체감)
3. docstring 을 모호하게 바꿔보고 AI가 도구를 잘못 고르는지 관찰. (설명의 중요성)
