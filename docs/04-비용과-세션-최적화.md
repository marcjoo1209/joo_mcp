# 04. 비용·세션 최적화 (Gemini + MCP)

> 대상: "API 세션을 유지하면 비용이 줄어드나? 메모리(캐싱) 기능은 뭐가 있나?"가 궁금한 사람

결론부터: **"세션 유지"와 "토큰(돈) 절감"은 다른 축**입니다.

| 비효율 | 줄이는 수단 | 절감 대상 |
|---|---|---|
| 요청마다 MCP subprocess 재기동 | **세션 유지(재사용)** | 지연·CPU·메모리 (토큰 비용 ❌) |
| 같은 prefix 토큰 반복 전송 | **컨텍스트 캐싱(메모리 기능)** | 입력 토큰 비용 ✅ |

---

## 1. 세션 유지 — 이 프로젝트가 실제로 한 것

### 기존 (요청마다 새로 띄움)
```
/chat 요청 ─► MCP subprocess spawn ─► initialize ─► list_tools ─► ... ─► 종료
/chat 요청 ─► MCP subprocess spawn ─► initialize ─► list_tools ─► ... ─► 종료   (매번 반복)
```

### 개선 (앱 수명 동안 1개 재사용) — `app/services/mcp_session.py`
```
앱 시작(lifespan) ─► MCP subprocess spawn ─► initialize ─► list_tools (1회)
/chat 요청 ─► (재사용) call_tool
/chat 요청 ─► (재사용) call_tool
앱 종료 ─► subprocess 정리
```

- **효과**: 요청마다 들던 프로세스 기동/초기화 지연이 사라짐. **토큰 비용이 아니라 지연·리소스 절감.**
- **주의(anyio 규칙)**: stdio 세션의 task group 은 *연 task 에서 닫아야* 한다.
  그래서 `lifespan` 안에서 `async with` 로 열고 닫고(같은 task), 요청 핸들러에서는
  `call_tool` 만 호출한다. 공유 세션이므로 도구 호출은 락으로 직렬화한다.

> 참고: Gemini `Client` 객체도 `ChatService` 안에서 1회 생성해 재사용한다.

---

## 2. "메모리 기능"으로 토큰 비용을 줄이는 법 — 옵션 정리

| 기능 | 무엇 | 효과 | 적용 조건 |
|---|---|---|---|
| **Implicit caching** | 동일 prefix 자동 캐시 (2.5 모델 기본 ON) | 반복 토큰 **약 75% 할인** | prefix ≥ **1,024토큰**(2.5 Flash) / 2,048(Pro) |
| **Explicit caching** (`client.caches.create`) | 시스템·도구를 캐시에 고정, TTL 지정 | 75% 할인 + 예측 가능 | 캐시 사용 시 system_instruction·tools 를 **후속 요청에 다시 못 넣음**(캐시에 포함). 큰 컨텍스트일 때 유리 |
| **Batch Mode** | 비실시간 비동기 처리(최대 24h) | **50% 할인** | 실시간 `/chat` 엔 부적합 |
| **Vertex AI Memory Bank** | 세션 간 장기 기억 자동 생성(개인화) | 대화 연속성 | GCP Vertex AI 스택 필요(AI Studio 키와 별개) |
| **대화 이력 직접 관리** | Gemini 는 stateless → `contents` 에 이력 누적 | 멀티턴 문맥 | 서버가 세션별 history 보관(직접 구현) |

핵심 제약: **캐시를 쓰면 `system_instruction`/`tools` 는 캐시에 넣어야 하고, 이후 요청엔 다시 전달하지 못합니다.**

---

## 3. ⚠️ 이 예제에서 캐싱이 "지금은" 무의미한 이유 (직접 측정)

이 프로젝트의 반복 prefix(시스템 프롬프트 + 도구 5개 정의)를 토큰으로 재보면:

```
시스템 프롬프트 + 도구 5개(JSON Schema)  ≈  744 토큰
```

캐싱 임계값(implicit: 1,024토큰)을 **넘지 못합니다.**

→ **현재 규모에선 캐싱으로 줄일 토큰이 사실상 0.**
   도구가 수십 개로 늘거나, 긴 시스템 프롬프트/RAG 컨텍스트가 붙어
   prefix 가 1,024토큰을 넘어설 때 implicit caching 이 자동으로 적용됩니다.
   (코드는 이미 반복 컨텍스트를 앞쪽에 두므로 그때 별도 수정이 거의 필요 없음.)

### 캐싱을 켤 만해지면 (참고 코드)
```python
# 시스템 + 도구를 캐시에 고정해 두고, 이후 요청은 캐시를 참조한다.
cache = client.caches.create(
    model=settings.gemini_model,
    config=types.CreateCachedContentConfig(
        system_instruction=SYSTEM_INSTRUCTION,
        tools=[gemini_tool],
        ttl="600s",
    ),
)
response = await client.aio.models.generate_content(
    model=settings.gemini_model,
    contents=message,
    # 캐시 사용 시 system_instruction/tools 는 여기서 다시 주지 않는다.
    config=types.GenerateContentConfig(cached_content=cache.name),
)
```

---

## 4. 우선순위 권장

1. **MCP 세션 재사용** — 적용 완료. 지연·리소스 절감(가장 실속 있음).
2. **컨텍스트 캐싱** — prefix 가 1,024토큰을 넘을 때 도입. 그 전엔 효과 없음.
3. **Batch Mode** — 실시간이 아닌 대량 작업이 생기면 50% 할인 고려.
4. **대화 메모리(멀티턴)** 가 필요하면 세션별 `contents` 이력 보관을 구현.
